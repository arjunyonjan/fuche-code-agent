🧠 Hybrid Architecture — Mapper + single_turn

Jun 13

Deterministic intent matching before LLM; LLM only for language, not execution.

User Input │ ├── commands::handle? ──→ /help, /cd, /status etc. │ ├── mapper::map(input)? │ │ │ ├── matched ──→ tools::run_command(cmd) ← deterministic, <1ms │ │ │ │ │ ▼ │ │ ollama::single_turn(prompt, result) │ │ │ │ │ ▼ │ │ LLM responds naturally ("Added text...") │ │ (no streaming, no tool loop, no spinner) │ │ │ └── unknown ──→ ollama::chat_stream(...) ← full agentic loop │ (streaming + tools + spinner) │ (Q&A / complex intents) │ ▼ User sees response

📋 Mapped Intents

open <file>→ xdg-open <file>
add <text> to <file>→ printf '...' >> <file>
cat / read <file>→ cat <file>
ls / list [path]→ ls -la [path]
search <pat> [in path]→ grep -rn 'pat' [path]
run <cmd>→ <cmd> (passthrough)
live-server→ npx live-server
anything else→ LLM agentic loop

🧰 Tool Architecture — Single Tool

Jun 13

Replaced 7 individual tools with 1 universal tool.

BUILD ✅ PLAN ❌

Before: search_web, read_file, write_file, edit_file, delete_file, list_dir, search_content

After: run_command — one tool to rule them all

📐 How It Works

  • 1. Model calls run_command with a shell command
  • 2. tools.rs:run_command() executes via sh -c "..."
  • 3. Captures stdout + stderr, returns result to model
  • 4. Gated by perms.execute — only in Build mode

📡 single_turn

Jun 13

Non-streaming, no-tools LLM call used after mapper execution.

  • 1. Mapper executes command deterministically → result
  • 2. ollama::single_turn() POSTS {"stream": false} with prompt + fake tool call + result
  • 3. LLM thinks it called run_command and got the result — responds in natural language
  • 4. No streaming, no tool loop, no spinner — ~500ms typical
